# Project retro — datasets

**Date:** 2026-05-18
**Scope:** The project end-to-end — downloaders (factbook, openalex, gutenberg, kaggle), the read-only FastAPI app, the OpenAlex normalization + FTS5 indexer, and the conventions we settled into. Five commits across about 28 hours of clock time, four of them landing changes and one a README.
**Status:** Working end-to-end on Tailscale for three of four sources. Factbook, OpenAlex (with author filter and FTS), and Gutenberg (metadata + raw `.txt` streaming) are all live. Kaggle remains a template that has never been exercised.

## Summary

Started as a folder of one-shot downloader scripts that drop datasets into per-source SQLite files under `data/<source>/`. Layered a thin read-only FastAPI app on top of those files, mounted on port 8002 behind Tailscale, with a generic `Page[T]` wrapper and one router per source. Then closed two gaps in the OpenAlex coverage that mattered for real use: author filtering required a backfill into normalized `authors` / `work_authors` tables, and useful discovery required an FTS5 index over `title` + `abstract`. The shape that emerged — independent scripts owning all schema/index/write concerns, the API owning only reads — held up cleanly through every phase.

## What went well

- **The "independent script per source" pattern paid off.** No shared library, no plugin abstraction, no config system. Each downloader sets its own table shape, indexes, and re-run semantics. Adding OpenAlex's normalization and FTS in later phases didn't touch the other sources at all. The cost — slight repetition of SQLite-open boilerplate — was much smaller than the cost of a premature framework would have been.
- **SQLite as the universal storage layer was the right call.** One file per source, trivially backed up by `cp`, no daemon, no auth surface. The read-only URI form (`file:...?mode=ro`) is the single feature that made the rest of the architecture work: module-level cached connections shared across FastAPI's threads, zero locking concerns, and a hard guarantee that the API can't corrupt anything.
- **Read-only API as a clean architectural boundary.** Every schema concern, every `CREATE INDEX`, every backfill lives in `scripts/`. The API is purely a query layer. When the OpenAlex author filter needed a new join table, the work happened in a script and the router just learned a new clause — no migrations, no coordination, no "is the API down while we reindex?" question.
- **`Page[T]` generic across all list endpoints.** Three sources, three filter shapes, one response contract. Adding `q`/`sort` to `/openalex/works` in phase 3 didn't change the wrapper at all. The generic was speculative when I wrote it; it earned its keep immediately.
- **Inline SQL in thin routers.** No ORM, no query builder, no migration framework. Each router is one file you can hold in your head, and the SQL is right there next to the filter logic. Pattern of "build `clauses` + `params` lists, join with AND" generalized cleanly from factbook (1 filter) through gutenberg (3) to openalex (7-ish).
- **CLAUDE.md as the persistent "what you'd need to know" doc.** Per-script gotchas, the index-creation convention, port reservation, the author-fragmentation limitation — all of it lives in one place, separately from the README that explains the API surface. Splitting these two audiences (collaborators vs. tooling/agents) kept both documents short.
- **Incremental phasing.** Download → API → author normalization → FTS, each shipped end-to-end before the next started. The phase-3 retro exists because phase 3 was a discrete unit. No half-finished branches, no "I'll wire this up after the next thing."
- **Param binding hygiene held everywhere.** No string-formatted user input made it into a SQL path across any router, including the FTS phase where it would have been tempting. The one place we accept FTS5 syntax in the user input, we still pass it as a bound parameter and let SQLite parse it.

## What went wrong / what I learned

- **OpenAlex authors-as-comma-joined-string was a schema mistake I had to backfill around.** The initial downloader stored author display names as a single `", "`-joined string, which made `/works?author=` impossible at the API layer without either a `LIKE '%...%'` scan (bad) or a normalization pass (what we did). The fix worked, but the cleaner fix would have been to keep OpenAlex's `authorships[].author.id` from the API response and write a real `work_authors` table from the start. That would have also avoided the credentialed-suffix fragmentation (~0.08% of works split into phantom rows on `", Jr."` etc.). **Takeaway:** when ingesting a foreign API, preserve its identity columns even if I don't think I'll need them. Strip later, never the other way around.
- **Operational errors keep surfacing as user errors.** Two examples now: `?q=...` before the FTS indexer has run returns "no such table: works_fts" as a 400, and `/health` reports per-DB errors in the body but always returns HTTP 200. Both are technically defensible but make remote debugging harder than it needs to be. A monitor pinging `/health` won't notice a broken database. **Takeaway:** when a precondition is operational (file exists, table exists, indexer has run), surface it as 503, not 400, and let the HTTP status reflect it.
- **Background process management for the dev server got noisy.** First flagged in the phase-3 retro and still true: `pkill ... && uvicorn ... &` chains tripped over themselves more than once. The mitigation — stop the server in its own command, confirm, then start it — is obvious in hindsight and I keep relearning it.
- **No automated tests of any kind.** Smoke-testing by `curl`-ing the running server worked for a project this size, but every refactor relied on me remembering the test matrix. The ambiguous-column-name bug in phase 3 was caught by a smoke test only because I had thought to write down a 7-axis matrix. A real test would have caught it at import time. **Takeaway:** at minimum, a `pytest` file that hits each route with one happy-path request would have a positive ROI now even if it didn't earlier.
- **`gutenberg_download.sh` isn't self-contained.** It rsyncs through a specific SSH host alias (`pop-os`) that won't resolve anywhere else. Fine for me, but it's a footgun the moment anyone else (or future me on a new machine) runs `bash scripts/gutenberg_download.sh`. Not worth fixing today, but worth noting in the script header.
- **`kaggle_download.sh` is dead code.** It's been in the repo since commit 1 and has never successfully downloaded anything; the template fields (`KAGGLE_USERNAME`, `KAGGLE_API_KEY`, `DATASET`) have never been filled in. Carrying it costs nothing, but it's worth being honest that "four downloaders" is really three plus a stub.
- **The retro convention isn't enforced consistently.** Phase 3 has a retro because I wrote one at the time. Phases 1 and 2 don't, and writing them now would be reconstructive. This document is partly making up for that. **Takeaway:** if a phase is large enough to be a discrete commit message, it's large enough to deserve a retro entry at the time, even a short one.
- **Connection caching has a subtle staleness mode.** `api/db.py` caches connections at module level, so if a downloader rewrites a DB while the API is running, the cached handle still points at the old file (or fails on reads from a truncated one). The docstring documents this; nobody has been bitten yet. But it means "rebuild the FTS index" effectively requires an API restart, even though both sides are technically fine in isolation.

## Decisions worth remembering

- **SQLite, not Postgres.** Right call for read-only, file-per-source data with infrequent rebuilds. No daemon to run, no auth to configure, backup is a file copy, the whole DB is portable across machines. The cost — no concurrent writers — is irrelevant because there is exactly one writer (the indexer) and it owns the file alone.
- **Module-level cached read-only connections, shared across threads.** Safe because of `mode=ro`; would not be safe otherwise. This single decision removed any need for a connection pool, dependency-injection scope, or request-scoped session management.
- **Port 8002 reserved, Tailscale ACLs as the only auth.** Matches the threat model (a private hobbyist API on a private network). No JWT, no API key middleware, no rate limiting. If this ever needs to be exposed publicly, the auth story is its own project.
- **Inline SQL, no ORM, no migration framework.** Read-only API; schemas are owned by scripts. Migrations would add more cognitive load than they remove.
- **External-content FTS5 for OpenAlex.** Index references rows by rowid, no content duplication, rebuild from scratch on each indexer run is fast enough (~20 s on 268k rows). Documented in detail in the phase-3 retro.
- **`INSERT OR REPLACE` / `INSERT OR IGNORE` everywhere.** Every script is safely re-runnable. No "delete the DB first" step in any runbook.
- **CLAUDE.md owns per-script gotchas; README owns the API surface.** Two audiences (agents / future-me vs. collaborators) get two documents, both stay short.

## Carry-over

- **OpenAlex authorship re-download.** The proper fix for the credentialed-suffix fragmentation (~0.08% of works) is a re-download that captures `authorships[].author.id` and builds `work_authors` from real identifiers. Deferred from phase 2; still deferred. Worth doing the next time the OpenAlex DB is refreshed for any other reason.
- **Gutenberg FTS.** Same shape as the OpenAlex phase-3 work, but the corpus is ~60 GB of `.txt` rather than ~750 MB of abstracts. Not a weekend project. Discussed in the phase-3 retro as a phase-4 candidate, not pulled in.
- **`/health` should reflect per-DB status in its HTTP code.** Currently always 200. Should be 503 when any database is broken. Small change.
- **No OpenAlex incremental top-up.** The current downloader is one big run. A second run would re-fetch everything. Adding a "fetch since last cursor" mode is the obvious next step the day this data starts to feel stale.
- **A minimal `pytest` smoke suite.** One happy-path request per route, plus the `relevance`-without-`q` 400 and the FTS5-syntax-error 400. Maybe 50 lines.
- **`kaggle_download.sh`.** Either fill it in for a real dataset or delete it. Carrying a stub forever isn't useful.
- **`gutenberg_download.sh` self-containment.** A header comment naming the required SSH host alias would be enough. Doesn't need to be made fully portable.
