# Arxiv full-HTML chunking — `html_to_markdown` port

**Date:** 2026-05-19
**Scope:** Smallest-valuable Phase 3 slice: port `html_to_markdown(html) -> str` from `local_wikipedia/arxiv/render.py` into `rag/render.py` (LaTeXML HTML → markdown; preserves math `alttext`, drops chrome), wire it into `scripts/arxiv_rag_extract.py` so `papers.html_content` is rendered when present (fallback to title+abstract otherwise), switch `scripts/arxiv_index_rag.py` to pass `chunk_fn=chunk_markdown` so the section headings from the rendered markdown populate `chunks.section`. Bump `CLEANER_VERSION` from `"v1"` to `"v2"` to force a full re-embed on next indexer run. Add 31 unit tests in `tests/test_render.py`. Defers the full Phase 3 ingest port (`oai.py` / `ingest.py` / `download.py` / author normalization) — those work against the existing copied `arxiv.db` and stay out of scope here.
**Status:** Committed as `cdb6f2a`, pushed to `origin/main`. 80/80 pytest pass (49 prior + 31 new render tests). One downloaded paper sanity-rendered end-to-end: 885 KB HTML → 79 KB markdown with 12 section headings (Abstract, Main idea and structure of the review, Magnetized turbulent astrophysical media, …). **No re-embed performed** — deferred to the user per WORK.md §3.8 / the established convention; the bumped `CLEANER_VERSION` makes the next `python scripts/arxiv_index_rag.py` run re-embed everything.

## Summary

Phase 2a embedded arxiv into RAG using `title + "\n\n" + abstract` only. That's ~1.3 chunks per paper and no section context. After this slice, papers with downloaded HTML get rendered to section-tagged markdown (one `##` heading per paper section), and the chunker switches from `chunk_doc` to `chunk_markdown` so each chunk carries its section label. Papers without HTML keep the current fallback path unchanged. Net effect on retrieval: `/arxiv/chunks` now operates over full paper bodies with section labels — a query like "experimental setup" can preferentially hit Methods chunks instead of just matching abstracts.

The slice deliberately ports one function (~270 lines, with helpers) and leaves the OAI/ingest piece of Phase 3 deferred. The migration plan originally treated all of Phase 3 as one block; recognising that `render.py` could land independently — against the existing `arxiv.db` copy — unlocked ~80% of the retrieval value for ~10% of the Phase 3 work.

## What went well

- **Plan-mode discipline.** Wrote `/home/kyle/.claude/plans/staged-enchanting-mountain.md` first, asked four clarifying questions (scope, fallback behaviour, tests, version bump), got user approval, then ran linearly through 10 tasks. No mid-stream rework. Recent retros keep calling out this pattern as positive and it keeps paying off.
- **Verbatim port over re-implementation.** Took `render.py` whole (`_render`, `_emit_math`, `_emit_heading`, `_render_list`, `_render_table`, `_inline_text`, `_cleanup`), dropped only the frontend-only `prepare_local_view`. Resisted the urge to "improve" anything while porting — the upstream is battle-tested, this slice's surface area is just integration. 31/31 tests passed on the first run.
- **One-shot sanity check against real data.** The 885 KB → 79 KB compression with 12 named sections is the expected shape; no debugging round-trip. The first downloaded paper happened to be an astrophysics review (`1108.2280`) with classic-shape headings — fortunate but representative.
- **Version-key augmentation as a side win.** Added an 8-char `html_marker` (first 8 chars of `content_hash(html_content)`) to the arxiv version key alongside `oai_datestamp` and `CLEANER_VERSION`. Means papers that get HTML downloaded *later* will re-embed on the next run, even when their OAI datestamp didn't move. Cheap defensive change; doesn't grow the diff much.
- **Smallest-valuable-slice instinct paid off.** The migration plan listed render as part of a multi-component Phase 3. Identifying it as the only piece that can land without an ingest schema change, and shipping it alone, is the move that makes the next phase optional rather than blocked.
- **Plan file was concrete enough to execute from.** Every file path, every change, every test case enumerated in the plan landed in the commit. No design surprises during implementation — exactly what a good plan file is for.

## What went wrong / what I learned

- **Plan's verification step #4 was deferred.** The plan called for a `--limit 30` indexer dry-run to measure chunks/doc and update CLAUDE.md's runtime estimate. Running it would have partially populated `data/arxiv/arxiv_rag.db` with v2 chunks alongside v1 chunks, plus would have required Ollama up. I skipped it pending user direction and noted CLAUDE.md says "re-measure after the next full run." The plan's verification list is now partially complete; the open item is procedural, not a code defect.
- **Almost forgot to update the SQL.** First pass at `arxiv_rag_extract.py` was the `iter_docs` logic only. On re-read I caught that the SELECT didn't include `html_content` in either the `--limit` branch or the no-limit branch. Lesson: when a change reads a new column, audit the SQL statement in the same edit, not a follow-up.
- **`CLEANER_VERSION` bump affects all four sources, not just arxiv.** Predictable but worth being explicit: the bump means factbook/openalex/gutenberg will also re-embed on their next indexer runs. The other three were already on the chunking-cleanup retro's carry-over list, so this isn't new work — it just makes the carry-over harder to defer.
- **One unfixed asymmetry in extractor output.** For HTML papers, the chunk body doesn't include the title (render.py drops the `<h1>`). For non-HTML fallback papers, the chunk body still includes `title + "\n\n" + abstract`. The title gets prepended at embed time via `format_document` regardless, so non-HTML papers end up with the title twice in the embedding payload. Considered cleaning this up; decided against to keep the diff scoped to "wire in render." Worth a one-line fix in a future commit.
- **Considered porting the upstream's `TestRealMistralFixture`.** Decided not to — would have required committing a 200 KB+ HTML fixture file, and the inline cases plus a real-data sanity render at the integration step gave equivalent confidence.

## Decisions worth remembering

- **Port verbatim; prune at the function boundary, not inside helpers.** Bringing all of `_render` and its helpers verbatim made the diff easy to audit (against the source) and meant test cases ported 1:1. Touching anything inside the helpers would have invalidated the upstream tests' shape and forced re-derivation.
- **Sentinel string for "no HTML" in the version key.** Used `"no-html"` literal as the marker for absent `html_content`, not an empty string or `None`. Explicit, grep-friendly, future-proof against the rare case where a real `content_hash[:8]` collides with empty string in some odd transformation.
- **Belt-and-braces fallback on render-empty-result.** `text = html_to_markdown(html_content).strip() if html_content else ""`, then `if not text: <fallback>`. Handles the edge case where `html_content` is non-null but the renderer produces nothing useful (all chrome, somehow). Two lines; no realistic-cost downside.
- **Tests inline-only, no fixture files.** Mirrors `tests/test_chunker_cleaner.py`. Fast (0.16 s for 31 tests), grep-friendly, no fixture management. Real-data validation lives at the integration verification step, not in unit tests.
- **`render.py` lives in `rag/`, not `scripts/`.** Same module level as `cleaner.py` and `chunker.py` — those are content transformers used by both the indexer and the API. `render.py` is the same kind of primitive; it just happens to currently have only one caller. Belongs alongside its peers.

## Carry-over

- **Re-embed arxiv against v2.** `python scripts/arxiv_index_rag.py`, restart uvicorn afterward. The earlier title+abstract-only estimate (~25-40 min) is a lower bound; full-body chunks will be several × the per-doc chunk count, so wall time is likely several × higher. **Re-measure on a small sample first** per WORK.md §1.10 before quoting an updated number in CLAUDE.md.
- **Re-embed openalex/factbook/gutenberg.** Already on carry-over from the chunking-cleanup retro; this slice's `CLEANER_VERSION` bump amplifies the pending work but doesn't add anything qualitatively new.
- **Render quality on edge-case papers.** Only validated against one real paper (`1108.2280`). Worth eyeballing one or two more during the re-embed: a recent heavy-math paper (does `alttext` survive everywhere?) and a pre-LaTeXML older paper (does the fallback degrade gracefully?). If either looks bad, file a follow-up.
- **Title duplication in non-HTML fallback.** Minor pre-existing redundancy now visible because the HTML path doesn't have it. A one-line cleanup in `arxiv_rag_extract.py` would unify the two paths: drop the `f"{title}\n\n{abstract}"` body composition in favor of just `abstract`, letting `format_document` carry the title in the embed prefix. Defer until next time the file's open.
- **Phase 3 proper.** This slice is the smallest-valuable cut; the full ingest port (`oai.py`, `ingest.py`, `download.py`, author normalization with structured `<keyname>`/`<forenames>`/`<affiliation>` per WORK.md §2.1) remains the next bigger code direction.
- **Empty `Carry-over` next time, ideally.** Per WORK.md §4.3, a one-line "Nothing carried over" is a positive signal. This list inherits the v2 re-embed work; the next retro should shed most of it.
