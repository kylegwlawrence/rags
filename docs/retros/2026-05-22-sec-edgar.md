# SEC EDGAR — body-fetch, FTS/RAG indexers, API routes, UI

**Date:** 2026-05-22
**Scope:** Build full-text + RAG search over SEC 10-K filings on top of the pre-existing metadata-only `sec_edgar.db`. New scripts under `scripts/sec_edgar/`: `sec_edgar_fetch_bodies.py` (standalone body-fetch into a new `body`/`status` column, decoupled from indexing per user request), `sec_edgar_index_fts.py` (`filings_fts` over company_name + body), `sec_edgar_rag_extract.py` + `sec_edgar_index_rag.py` (flat `chunk_doc` RAG). API: `db.py` connections, `SecEdgarFiling` model, `api/routers/sec_edgar.py` (list / detail / content / chunks / doc-chunks), router mount + `/health`. Frontend: one `sources.js` entry. Docs: CLAUDE.md. 10-K only, newest-first, small default caps.
**Status:** Committed `a3624a2` (feature), `a0d2888` (schema fix), `4cede92` (iXBRL strip). Not pushed. Verified end-to-end live: metadata 3,056,588 filings (1993-08-13 → 2026-05-21, 277,536 10-Ks); 25 newest 10-K bodies fetched with iXBRL stripped; FTS over all 25 (`q=climate` → 20 hits); RAG over 3 filings (1,052 chunks) with relevant semantic hits. `/chunks` + `/doc-chunks` confirmed after uvicorn restart.

## Summary

The repo already had a `sec_edgar_download.py` that harvested filing *metadata + URLs* but no document text — unlike every other RAG source, which indexes stored text. So the build had to add a text-acquisition step before any indexing was possible. We mirrored the `github_readmes` pattern (metadata rows + a later body-fetch gated by a `status` column) and the `federal_register` router/indexer wrappers, which made the API/frontend/RAG wiring near-mechanical. The user's one explicit design ask: keep body-fetch a standalone step that does **not** trigger FTS or chunk/embed.

The headline event of the build wasn't the new code — it was discovering the existing downloader was silently corrupt. A staged "verify the raw data before trusting the pipeline" approach surfaced it on the first body-fetch (5/5 errors). Two more issues (a schema gap, iXBRL noise) were each caught at the right moment by inspecting actual output before committing to expensive steps.

## What went well

- **Pattern reuse paid off, again.** Cloning `github_readmes` (status-gated body column) and `federal_register` (router + `run_indexer` wrapper) meant the frontend needed **zero** component changes — just a `sources.js` entry — and the router/indexer were assembly, not design. Same dividend the recent retros keep noting.
- **Staged verification caught the big defect early.** Sanity-checking the stored rows before running the pipeline exposed that every `filing_url` 404'd. Had we trusted the downloader and gone straight to a large fetch+embed, we'd have burned hours embedding nothing (or garbage). "Verify raw data first" is the lesson worth carrying to the next source.
- **Caught iXBRL noise before the expensive RAG run.** Inspecting `/content` showed modern 10-Ks lead with thousands of hidden XBRL tagging facts (~6% of body). Fixing extraction *before* embedding 3 filings meant the embeddings were clean on the first real RAG run, not after a re-do.
- **Asked about scale instead of assuming it.** 277K 10-Ks × ~250 chunks is an overnight-plus job. Stopping to ask "how many for this run" (answer: 3) avoided committing the machine to hours of embedding during a verification pass.
- **Plan-mode discipline held.** Wrote the plan, asked the up-front questions (body strategy, form types, scope cap), got approval, executed linearly. The mid-build pivots (corrupt data, scale) were surfaced as questions, not silent decisions.

## What went wrong / what I learned

- **The pre-existing downloader was silently broken.** `parse_index` used fixed-width column slicing trusting the `company.idx` *header* positions, but the data rows sit ~5 columns right of the header — so every `date_filed` truncated (`2025-02-13` → `2025-02`) and the date tail corrupted every filename/URL. All ~2.9M original rows were affected. Fixed with a structure-anchored regex (form has no spaces, CIK is digits, date is `YYYY-MM-DD`, filename is `edgar/…txt`). **Lesson:** an upstream component "already in the repo" is not a verified component.
- **A schema gap stranded the live API.** `body`/`status` existed only as a lazy `ALTER TABLE` in the body-fetch script, so a fresh `--reset` download left `/sec_edgar/filings` erroring on `no such column: filings.status`. Moved the columns into the canonical `create_schema`. **Lesson:** if the API hard-depends on a column, that column belongs in the source-of-truth schema, not a downstream migration.
- **`--reset` + cached connections = stale reads.** Rebuilding `sec_edgar_rag.db` (unlink + recreate → new inode) left uvicorn's cached read-only connection on the dead inode; `/chunks` served the old test index until restart. Known behavior (CLAUDE.md says restart after a rebuild), but it presented confusingly as "endpoint works but wrong companies." In-place re-indexes without `--reset` are picked up via WAL; file replacement is the case that needs the restart.
- **The "bad query" error masked a schema error.** `no such column` came back as HTTP 400 "bad query" (it's an `OperationalError` that `is_operational_error` doesn't classify as operational). Misleading, but the root-cause fix (schema) made it moot, so left the classifier alone.
- **iXBRL strip needed care not to over-remove.** Visible financial figures are wrapped in `<ix:nonFraction>`/`<ix:nonNumeric>` tags in the narrative; only the hidden `<ix:header>` / `display:none` block is noise. Decomposing *all* `ix:` tags would have deleted real numbers — so the strip targets only the hidden container.

## Decisions worth remembering

- **Body-fetch is a standalone step, on purpose.** Per the user: runnable without triggering FTS/RAG. Three independent scripts (fetch, FTS, RAG) over one combined pipeline — matches the repo's per-step convention and lets coverage grow incrementally.
- **List endpoint surfaces only `status='fetched'`.** Every browsable filing has content + (eventually) chunks; the 277K metadata-only rows stay hidden until their bodies are fetched. Cleaner UX than exposing 3M rows where 99.99% have no content.
- **Structure-based parse over fixed-width.** The regex is robust to column drift across 33 years of index format; the original fixed-width approach was the root defect.
- **iXBRL stripping lives in the fetch script, not shared `rag.cleaner.strip_html`.** It's SEC-specific (`ix:header` knowledge); the shared cleaner stays generic. Per the repo convention that cleanup decisions live close to the source.
- **Flat `chunk_doc` at 1500/1800/150 for now.** Filing text is treated as flat prose; deliberately deferred structure-aware chunking (see carry-over).

## Carry-over

- **Revisit SEC chunking & extraction (user-flagged).** Current chunking works but is crude. Levers identified: (1) structure-aware split on 10-K "Item N." headings so chunks carry a `section` like "Item 1A. Risk Factors"; (2) strip/prose-ify tables, exhibit indexes, signature blocks, and stray XBRL doc-id tokens (e.g. a leftover `wms-20260331` at body start); (3) re-tune chunk size — filings are long/dense, 1500 may be small. Also saved to project memory (`project_sec_edgar.md`).
- **Body-fetch: distinguish 404 → `missing` vs transient → `error`.** `fetch_body` currently returns `None` for both, so a genuinely-gone filing is bucketed as `error`. Harmless today (both terminal/skipped on re-run) but muddies any future "retry the errors" logic.
- **Broaden form-type match for legacy variants.** `extract_primary_document` matches `<TYPE>` exactly equal to the form, so `10-K405` / `10-KSB` (pre-2003) fall back to the first document. Fine for newest-first current data; revisit when backfilling old years.
- **Scale the corpus when ready.** Only 25 bodies fetched / 3 embedded. `sec_edgar_fetch_bodies.py --limit N` → `sec_edgar_index_fts.py` → `sec_edgar_index_rag.py --limit N` (incremental, no `--reset` needed; skips already-embedded docs), restart uvicorn. RAG runtime ≈ 6 min/filing — plan accordingly.
- **Push the three commits** when the branch is ready (currently local-only on `main`).
