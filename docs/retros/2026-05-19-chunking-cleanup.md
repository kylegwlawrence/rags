# Chunking-and-embedding cleanup — v1 cleaner refactor

**Date:** 2026-05-19
**Scope:** Replace the regex chunker in `rag/chunker.py` with a recursive boundary-aware splitter built on `langchain-text-splitters`; add `rag/cleaner.py` (HTML stripping via `beautifulsoup4`, markdown-syntax stripping, whitespace normalisation including CRLF folding); wire per-source HTML/markdown/banner cleanup into the four extractors (`scripts/{arxiv,openalex,factbook,gutenberg}_rag_extract.py`); add `--max-chunk-size` hard-cap CLI flag plus per-source `--chunk-size` defaults (factbook 1000/1200, arxiv 1500/1800, openalex 1500/1800, gutenberg 2000/2400); invalidate every stored doc via a `CLEANER_VERSION = "v1"` suffix on each `Doc.version`.
**Status:** Code shipped, all 49 tests pass (14 new chunker/cleaner + 35 existing smoke). **No re-embed performed** — re-embeds deferred to the user with explicit resume commands per WORK.md §3.8. Existing `_rag.db` files still contain dirty pre-refactor chunks until the user runs each `*_index_rag.py` again. Awaiting commit approval per WORK.md §1.2.

## Summary

Audited 1,390 chunks across the four `_rag.db` files. Found four classes of problem degrading retrieval: HTML tags leaking through (factbook 26.8% had `<br>`, openalex 78 chunks with `&amp;`), mid-word truncations (arxiv 8.4% / openalex 13.4% / factbook 38.2% / gutenberg 28.6% of chunks started with a lowercase letter), Project Gutenberg banner remnants in 5/14 gutenberg chunks, and stray markdown syntax in the embed payload. After the refactor, on a 30-doc-per-source resample: zero HTML, zero entities, zero banner remnants, zero mid-word starts in arxiv/openalex/factbook, **6.7%** lowercase starts remaining in gutenberg (down from 28.6%; the remaining 2 of 30 are PG metadata paragraphs adjacent to banners where the banner regex cleaned the line containing "Project Gutenberg" but left explanatory prose around it).

The user explicitly chose `langchain-text-splitters` + `beautifulsoup4` over a stdlib reimplementation. CLAUDE.md "prefer stdlib" is now annotated to record this deviation and its rationale.

## What went well

- **One-PR-fixes-all-four (WORK.md §2.9).** The same audit pattern applied to four extractors meant the fix needed to land in all four together. Project history has at least one case (Phase 2 503-vs-400 fix in arxiv that missed openalex) where the same shape leaked from one peer to the next; this refactor avoided that by editing four extractors in one pass. The shared `rag/cleaner.py` made the per-extractor change ~5 lines each, so the fix was cheap to peer-replicate.
- **Cleaner-version invalidation pattern.** Suffixing every `Doc.version` with `CLEANER_VERSION` makes "the cleaning logic changed, re-embed everything" a one-line change that propagates through the existing version-hash skip in `run_indexer`. No new CLI flag, no force-rebuild, no `--reset` required. Bumping `CLEANER_VERSION` to `"v2"` later will Just Work the same way.
- **Hard-cap post-pass under the soft target.** Langchain's `RecursiveCharacterTextSplitter` is soft — it can overshoot. The post-pass that re-splits anything over `max_chunk_size` with a tighter `[" ", ""]` splitter caught the long-paragraph-no-punctuation case (the 5,000-char no-break test passes) without compromising natural boundaries on normal text.
- **CRLF normalisation in `normalize_whitespace`.** First gutenberg audit showed 16.1% lowercase starts after the chunker rewrite. Inspecting the leaks showed `\r\n` line endings — gutenberg uses Windows-style endings — were preventing the `\n\n` paragraph separator from firing, so the splitter fell through to single-newline boundaries mid-sentence. Folding CRLF into the whitespace normaliser dropped that to 6.7% with no source-specific code.
- **Pre-flight measurement caught the wrong runtime estimate (WORK.md §1.10).** First batch-of-32 latency test used `'sample text'` strings (~10 chars) and got 75 ms/chunk. Re-running with realistic ~1600-char content gave 1.0-1.9 s/chunk — same shape as the prior retros' 1 s/chunk. Without the second pass, the runtime estimate would have been 18× low. The measurement step lives in CLAUDE.md as a `before quoting an estimate, re-measure` reminder; the rule from WORK.md §1.10 keeps paying off.
- **Plan-mode-then-execute discipline.** The user toggled into plan mode, asked clarifying questions, then approved the plan before any code changed. That kept the 13-task implementation linear with no rework — each step was already justified by the plan, so the editor could move through them in order.
- **Existing smoke tests stayed green.** 35 existing tests against on-disk `_rag.db` files (still containing the dirty pre-refactor chunks) passed unchanged. The API contract didn't break; only the indexer-side text pipeline changed. That's the right boundary — read-only API doesn't care about chunk hygiene, only schema correctness.

## What went wrong / what I learned

- **First batch-latency measurement used unrealistic content.** The 32-batch test with `'sample text'` × 32 returned in 2.4 s (75 ms/chunk). Plugged into the runtime estimator that gave **1.3 minutes** for the full arxiv corpus — obviously wrong. Re-ran with ~1600-char strings and got 31-60 s per batch (1.0-1.9 s/chunk), matching the prior retros. The lesson: when measuring per-unit cost, the unit's *size* matters as much as the *count*. Use real-shaped inputs from the start. This is a refinement of WORK.md §1.10 worth recording.
- **Trial-1 latency was ~2× trial-3.** Cold-start variance on Ollama: first batch 60.7 s, second 39.5 s, third 31.3 s. The mean (1.37 s/chunk) overstates the steady-state rate; the warmed value (~1.0 s/chunk) is closer to what the long re-embed will sustain. CLAUDE.md quotes a range to reflect this rather than a single number.
- **`MarkdownHeaderTextSplitter` collapses `\n\n` to `  \n`.** Langchain's MHTS converts paragraph breaks inside sections into markdown "soft line break" syntax (two spaces + newline). The chunker passes the output back through `normalize_whitespace`, which collapses the horizontal whitespace. Without that post-pass, factbook chunks would have had visible `  \n` artifacts in the embed payload. Caught it during the first smoke test before the audit.
- **Test assertion error: `&lt;` decodes to literal `<`.** `test_strip_html_decodes_bare_entities` originally asserted `"<" not in strip_html("a &lt; b")` — wrong, because `&lt;` correctly decodes to a literal `<` character (which is what we want for an embedding payload). Fixed the test to assert `"&lt;" not in result` instead.
- **Factbook "urban: urban:" duplication is source data, not a walker bug.** Investigated per the plan's open question. Confirmed: factbook JSON has key `"improved: urban"` with text value `"urban: 99% of population..."` — so the rendered line is `"...improved: urban: urban: 99% ..."`. Not a `_walk_to_lines` bug. Left as-is and documented the caveat in the CLAUDE.md factbook script note.
- **Gutenberg banner regex still misses some paragraph context.** 2 of 30 gutenberg chunks (6.7%) start lowercase because the `_PG_MENTION_LINE_RE` defensive scrub only removes lines that contain "Project Gutenberg" by name. Adjacent paragraphs that describe PG history without using the name (e.g. "and now officially re-released on November 19, 1993...") survive. Fixing this would require paragraph-level scrubbing of the pre-text-1 metadata, which risks eating legitimate content. Accepted at 6.7% (down from 28.6%) and recorded here.
- **One transitive-dependency expansion.** `langchain-text-splitters` pulled in `langchain-core`, `langsmith`, `tenacity`, `jsonpatch`, `uuid-utils`, `xxhash`, `zstandard`, `orjson`, `requests-toolbelt`, and `langchain-protocol`. About 6 MB of extra deps. Worth flagging — the choice of `langchain-text-splitters` was made explicitly knowing the tradeoff, but the transitive footprint is heavier than the splitter package alone suggested.

## Decisions worth remembering

- **Embed plain text, keep section as plain prefix.** The user asked whether to embed markdown or plain text. Decision: plain text. `nomic-embed-text:v1.5` is a BERT-family model trained on natural prose; raw `##`, `**`, `[link](url)` tokens displace useful content. Section context survives via the `section` column and `embedder.format_document`'s `"{title} - {section}\n\n{text}"` prefix — that's plain text, no markdown syntax. So the chunker strips `##` from chunk bodies into the section field, and the embed payload carries section text but no markdown markers.
- **Per-source chunk-size policy.** Factbook is dense key:value (1000/1200). Arxiv and openalex are short abstracts (1500/1800). Gutenberg is narrative prose (2000/2400). Each indexer script exposes both `--chunk-size` and `--max-chunk-size` so per-corpus tuning stays cheap.
- **Cleanup lives in extractors, not the shared indexer.** Each source has a different idea of what's noise (factbook walks JSON leaves; gutenberg has PG banners; arxiv abstracts are mostly clean). Putting `strip_html`/`strip_markdown` into the per-source extractors keeps the shared indexer source-agnostic and avoids `if source == "factbook"` branches.
- **`langchain-text-splitters` over a stdlib reimplementation.** The user weighed this against CLAUDE.md "prefer stdlib" and chose the dependency because the audit found 8-38% mid-word cuts across sources — a stdlib reimplementation would have had to reproduce langchain's recursive boundary search, which is ~100 lines of careful code. Buying the abstraction was worth the dep cost. Recorded in CLAUDE.md as an explicit "Conventions" deviation.
- **`beautifulsoup4` over stdlib `html.parser`.** Same reasoning. The factbook JSON had embedded `<br>` and `<p>` in 26.8% of leaves; `BeautifulSoup(t, "html.parser").get_text()` is one line and well-tested. Stdlib subclassing of `HTMLParser` would have been ~30 lines. Recorded as deviation.
- **`CLEANER_VERSION` as the invalidation knob.** The version-hash skip in `run_indexer` already handles "skip docs whose source data hasn't changed." Appending `CLEANER_VERSION` to every `Doc.version` extends this to "also skip if the cleaning logic hasn't changed." Bumping the constant becomes the canonical way to ship a re-embed without forcing `--reset`.
