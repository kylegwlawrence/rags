# Phase 2 overall retro — RAG across four sources

**Date:** 2026-05-18
**Scope:** All of Phase 2 (2a arxiv → 2b openalex → 2c factbook → 2d gutenberg) plus the state of the project end-to-end as of commit `7e2f800`. Sits alongside `2026-05-18-project-retro.md` (written this morning before Phase 2 began) and the four sub-phase retros. Reads as a "what did Phase 2 teach us, and where does the project sit now" companion to the per-sub-phase detail.
**Status:** Phase 2 shipped. Four `/<source>/chunks` endpoints live, each backed by a per-source `_rag.db` file with the uniform `chunks` / `chunks_fts` / `chunks_vec` / `docs_meta` / `_meta` schema. Tests at 30/30 green. Three of the four corpora are deferred mid-embed (openalex at 2620/5000, factbook at 50/261, gutenberg at 3/100) — the endpoints work against partial corpora and resume non-destructively.

## Summary

Phase 2 added one capability — hybrid (FTS5 + sqlite-vec) chunk search — across four heterogenous data shapes (flat columns, sampled flat columns, nested JSON, filesystem `.txt`). It landed in four sub-phases over ~10 hours of clock time. The shared `rag/` package emerged in 2a, gained a `chunk_fn` extension point in 2c when factbook needed markdown-aware chunking, and absorbed gutenberg in 2d without internal conditionals. The single biggest design choice was refusing to special-case per source in the shared code; instead, per-source variation is captured entirely in the extractor closure passed to `run_indexer`. That choice held against four genuinely different sources and would absorb a fifth without modification.

The project ends Phase 2 with eight SQLite files under `data/`, three deferred embed runs (a system-level GPU question, not a code question), and a code review's worth of follow-ups documented separately in `docs/code-review-2026-05-18.md`.

## What went well

- **Refactor-one-phase-early was the right call.** Phase 2c extracted `rag/indexer.run_indexer` and the `add_chunks_route` factory and the parametrized chunks tests, all at once, before factbook landed. The alternative — extract after gutenberg — would have meant rewriting two more sources post-hoc. `WORK.md` §3.6 says wait for three; in practice the third source on deck plus two existing examples gave enough signal. The rule should read "extract when you have three examples *or* two plus a clear third specced out."
- **Per-source variation lives in the extractor closure.** No conditionals inside the shared indexer skeleton, no `if source == "factbook":` branches anywhere. The extractor for each source is 30–110 lines and is the only file that knows anything source-specific. New sources will be a new extractor + a ~50-line indexer wrapper + one `RAG_SOURCES` line in tests — no edits to shared code.
- **Generic table names (`chunks`, not `<source>_chunks`).** Source identity comes from the file path; the SQL is identical across all four `_rag.db` files. This made the retriever genuinely shared — one function, four callers, no fanout.
- **`chunk_fn` plumb-through pattern.** Default `chunk_doc` handles flat text (arxiv, openalex, gutenberg). Factbook passes `chunk_markdown`. Future sources can pass any `(doc, *, chunk_size, overlap) -> list[dict]` callable. The signature didn't change between 2c (introduction) and 2d (reuse).
- **Defer-the-embed is a legitimate ship state.** Three of the four corpora are partial. Each endpoint still works, just over a smaller corpus. The version-hash skip in `run_indexer` makes resume idempotent — the user re-runs the same script with no `--reset` and it picks up at the next unembedded doc. This is the right answer to "the embed takes hours and the endpoint is independently useful."
- **Sparse-only fallback when Ollama is unreachable.** The `/chunks` endpoint returns 200 with `used_dense=False` when the embedder times out. The client gets degraded results plus a clear signal. Caught regressions immediately in the parametrized `test_chunks_sparse_only_when_ollama_down`.
- **Parametrized chunks tests.** One `pytest.param` line in `RAG_SOURCES` adds six test runs for a new source. Scaled 1→2→3→4 with zero test maintenance per source. The `pytest.skip` fallback on empty corpora kept the suite green during the deferred embeds.
- **Schema-mismatch auto-rebuild.** Detects `paper_chunks*` (legacy upstream) and embed-model/dimension mismatch, wipes the file, and rebuilds with a stderr note. No `--reset` ever needed for the obvious cases.
- **Pre-flight chunk-count estimates.** Started missing in 2a (silent 14-minute run with no progress signal), inverted in 2b (added flush-boundary prints), and became standard practice in 2c/2d (count chunks before estimating runtime). The retros document several wildly wrong eyeball estimates that pre-flighting would have caught: 6× off in 2b, 12–24× off in 2c, ~20× off in 2d.
- **Shared `is_operational_error` helper.** Lives in `rag/retriever.py`. Used by the retriever (decides swallow vs. propagate) and by routers (decides 503 vs. 400). Centralized the "missing table vs bad SQL" classification in one place. Inconsistently applied — see the corresponding code-review finding — but as a primitive, correct.
- **`content_hash` helper for version keys.** Sources without an `updated_at` column (openalex, factbook content, gutenberg files) all need a content-derived change-detection key. The shared helper handles the common case; factbook deliberately inlines its single-arg hash to avoid invalidating the previously-stored versions when the helper was added (one-line comment explains it).
- **File-fingerprint version key for filesystem-backed sources.** Gutenberg uses `{size_bytes}-{sha256_prefix_of_first_and_last_4kb}`. Stable against rsync-touched mtimes; doesn't require reading the whole file. The right shape for any future source where the body lives on disk.

## What went wrong / what I learned

- **Embed-runtime estimates were wrong by big multiples in every sub-phase.** 2b: 6× off (5–10 min predicted, ~60 min actual). 2c: 12–24× off. 2d: ~20× off. Same lesson each time — eyeballed multiplications keep losing. The fix that finally stuck was "count chunks against real data, then multiply by the measured per-chunk rate from the prior sub-phase." This belongs in `WORK.md`.
- **CPU is the embedding bottleneck.** `nomic-embed-text:v1.5` on local Ollama runs at ~1 s per 1600-char chunk on CPU. Phase 2 produced ~50,000 chunks worth of work; the three deferred embeds aggregate to many hours. GPU acceleration (Apple Metal, AMD ROCm, NVIDIA CUDA) is the highest-value perf lever the project has. Whether one is available locally has been the unstated load-bearing question for the last three sub-phases.
- **The 503-vs-400 fix for FTS missing-table didn't get backported to openalex.** Arxiv's `list_papers` correctly classifies via `is_operational_error`; openalex's `list_works` does not. Same SQL shape, asymmetric error handling. This is exactly the failure mode `WORK.md` §2.4 warns about for ambiguous columns (audit peer code in the same edit) but the rule's current wording is JOIN-specific. The lesson generalizes: when fixing a bug in one place, audit the parallel implementations in the same change.
- **Five carry-over items have aged across five retros without action.** `/health` HTTP status (200 vs 503), connection-cache staleness, `kaggle_download.sh` stub, openalex authorship re-download, and lazy-load `html_content`. Each is documented, each is small, none have happened. The presence-of-carry-over signal is working; the resolution loop isn't. WORK.md should either set a TTL ("if it shows up in three retros, decide: do it or accept it formally") or accept that some carry-overs are permanent and move them to a `KNOWN_LIMITATIONS.md` so they stop crowding the active list.
- **Two old downloader scripts (`factbook_download.py`, `openalex_download.py`) are noticeably looser than everything else.** No `__main__` guard, no `mkdir -p`, top-level mutable state, no retries on HTTP errors. They predate Phase 1 and inherited none of its conventions. The retro pattern keeps surfacing this ("dead code", "not self-contained") but never with an action.
- **`requests` is imported but not in `requirements.txt`.** Surfaced during this review. Only `openalex_download.py` uses it; the rest of the project uses `httpx`. Either port to httpx or add the dep — current state is "works on my machine."
- **Indexer's final-flush + FTS-rebuild step isn't transactional.** If the very last flush succeeds but the FTS rebuild fails, partial chunk data is durable but `chunks_fts` is stale. Documented in the phase 2b retro; still true. Not a blocker because re-running the indexer rebuilds FTS at the end of the next run.
- **Test data is implicit.** The smoke suite assumes each `_rag.db` has at least one chunk that matches "learning". This was a deliberate choice to keep tests against real data, but it means CI without prebuilt DBs would just skip everything. Acceptable for a personal repo; would need fixtures for any shared CI.

## Decisions worth remembering

(Decisions specific to Phase 2 — see `2026-05-18-project-retro.md` for the broader read-only-API, SQLite-as-storage, inline-SQL decisions.)

- **`rag/` at the repo root.** Not under `api/` (would force scripts to import across the boundary) and not under `scripts/_lib/` (would force the API to do the same). Peer to both.
- **One uniform RAG schema across sources.** `chunks(chunk_id, doc_id, section, chunk_index, text, text_length)` + `chunks_fts` + `chunks_vec` + `docs_meta(doc_id, version, title, chunk_count, indexed_at)` + `_meta(key, value)`. Same DDL on every `<source>_rag.db`. Source identity from the file path, not the table name.
- **`EMBED_MODEL = "nomic-embed-text:v1.5"`** — specific tag, not bare. Stored in `_meta.embed_model` so the indexer wipes-and-rebuilds on model change rather than silently mixing vectors from different models.
- **Per-source `chunk_size` is configurable.** Defaults to 1600 for abstracts; gutenberg uses 2000 because plain-text prose is denser than abstracts. The shared chunker accepts the parameter; the indexer plumbs it through.
- **`Page[T]` for paginated lists; `ChunksResponse` for RRF hybrid search.** Different contracts. RRF doesn't paginate; `used_dense` doesn't fit in `Page`. Adding the separate model in 2a was the right call.
- **Sparse-only fallback on Ollama failure** (`used_dense=False`, 200), not 503. Clients can decide whether to retry. Tested explicitly via monkeypatch.
- **503 for missing index tables / unreadable DB files; 400 only for empty `q` or FTS syntax errors.** Routers translate `sqlite3.OperationalError` through `is_operational_error` to distinguish. (Currently inconsistent on the openalex non-RAG router — see code review H2.)
- **Extractor returns `Iterator[Doc]`.** Not a list — sources can be arbitrarily large. The indexer's batched flush bounds memory regardless of source size.
- **`chunks_vec` rows deleted before `chunks` before `docs_meta`** in `flush()`. sqlite-vec virtual tables don't participate in FK cascade; explicit cleanup. Covered by `test_rag_no_orphan_vectors` so future refactors can't break it silently.
- **Mid-run pause is supported.** Flush-per-batch transactions are durable. Version-hash skip on resume picks up where it left off. Final FTS rebuild only runs at end of a full pass — degraded (dense-only) state until then. Documented in `CLAUDE.md`.
- **Refactor with the next source in mind, not after it.** Stronger form of `WORK.md` §3.6. When the third example is clearly on deck and the first two are nearly identical, extracting one phase early is correct. (Phase 2c, before 2d.)
- **Defer-the-embed (or defer-the-bulk-job) is a ship state.** Endpoint exists, smoke tests pass, schema is correct, a sample of N docs validates the pipeline. The corpus grows at the user's discretion via the same idempotent script. Not "half-finished" — fully shipped against a smaller scope.

## Carry-over

- **Resume the three deferred embeds.** Openalex (2380 docs remaining), factbook (211 countries), gutenberg (97 books). All three resume via the same idempotent indexer script with no `--reset`. Estimated CPU runtime: ~25 min + ~85 min + ~9 hours respectively. The right gating question is whether GPU acceleration arrives first.
- **GPU acceleration as a system-level project.** Three deferred embeds aggregate to ~10 hours on CPU. Checking what local GPU exists (`lspci`, `system_profiler`, etc.), enabling Ollama to use it (Metal is zero-config on macOS; ROCm and CUDA need driver work), and running one timing test would answer whether this is worth pursuing. Highest-value perf investigation in the project.
- **Phase 3: arxiv OAI ingest port + `render.py` + full-HTML chunking.** Promotes arxiv from title+abstract chunking to full-HTML chunking. `chunk_markdown` is already in place from Phase 2c; the missing piece is the HTML→markdown render step from `local_wikipedia`. After this, `arxiv/` becomes removable from `local_wikipedia` (which is the original motivation per `docs/arxiv_migration_plan.md`).
- **Code-review follow-ups.** `docs/code-review-2026-05-18.md` enumerates four High, six Medium, and seven Low/style items. The two most worth doing soon are H1 (`requests` not in `requirements.txt`) and H2 (the 503/400 fix not backported to the openalex non-RAG router). H3 (`/health` returning 200) has now appeared in six retros' carry-overs — either action it or move it to a permanent-acceptance document.
- **OpenAlex full-corpus embed.** 268k works × ~0.7 s/doc on CPU = ~50 hours. Top-5k is the deliberate Phase 2b scope decision; full corpus is "decide when there's a use case."
- **Cross-encoder re-ranking as a possible third retrieval stage.** Current `/<source>/chunks` flow is two-stage: FTS5 + sqlite-vec each surface up to `candidate_k` (default 50) chunk_ids, then RRF (Reciprocal Rank Fusion, k=60) merges them by rank position alone and returns the top `top_k`. RRF is tuned for *recall* ("did the right chunk land anywhere in the top 50?"); it doesn't know which of those 50 are actually best for the query. A cross-encoder re-ranker is the standard third stage: take the ~50 RRF candidates, score each `(query, chunk)` pair with a small transformer that lets every query token attend to every chunk token, re-sort, return the top K. This is the *precision* stage. Distinction matters because the stage-1 retrievers are bi-encoders — query and chunk are embedded separately and compared via cosine, so the model never "sees" them together; a cross-encoder does, and catches fine-grained mismatches like "chunk mentions auroras but is actually about Jupiter's magnetosphere." Cost is the reason it's not stage 1: cross-encoder scores aren't pre-computable, so it runs N forward passes per query (one per candidate) — fine over 50, hopeless over 1.6k or 268k. Implementation hook is `rag/retriever.py` only; no indexer changes needed. Concrete local options: `BAAI/bge-reranker-v2-m3` (~568 MB, ~20–50 ms/pair on CPU → ~1–2 s for 50 candidates) or `mixedbread-ai/mxbai-rerank-xsmall-v1` (~70 MB, faster, slightly worse quality). Ollama doesn't run cross-encoders natively today, so this pulls in a PyTorch dependency via `sentence-transformers` or `FlagEmbedding` — meaningful weight against the "no shared library" repo character. Gating signal: re-ranking earns its keep when (a) the corpus is large enough that "top 50 from RRF" reliably contains the right chunk but in the wrong position (mostly true at openalex/gutenberg scale, less true at arxiv/factbook scale), (b) queries are natural-language questions rather than keyword lookups, and (c) the consumer cares about top-3/top-5 specifically rather than browsing top-20. The most likely first place to add it is a future `/<source>/answer` endpoint that picks the *single* best chunk to feed an LLM — that's where the precision gap between "20 plausible results" and "1 best result" actually shows up. The existing `candidate_k` query parameter (already exposed, default 50, max 200) is the tuning knob.
- **Older Project Gutenberg banner variants** — the regex catches modern `*** START/END OF (THIS|THE) PROJECT GUTENBERG EBOOK ... ***` only. Older files and Small-Print sections slip through. Not blocking; revisit if a corpus expansion includes pre-1995 files.
- **OpenAlex authorship re-download** — original carry-over from the project retro. The ~0.08% credentialed-suffix fragmentation is still present. Proper fix uses `authorships[].author.id` from the OpenAlex response. Worth doing the next time the OpenAlex corpus is refreshed for any other reason.
- **Connection-cache invalidation** — still a uvicorn-restart-after-rebuild workflow. See code review M6 for the disposition options. The current limbo (documented carry-over for five retros, no movement) is the worst state.
- **Lazy-load `html_content` in `/arxiv/papers` list** — original carry-over from phase 1. Not in any hot path yet.
- **`kaggle_download.sh` disposition** — fill in, delete, or annotate. Carry-over from the project retro; still a stub. M5 in the code review.
- **Empty `Carry-over` next time, hopefully.** Per `WORK.md` §4.3, a one-line "Nothing carried over" is a positive signal. This retro's list is long because it consolidates five retros worth of unresolved items; the next phase retro shouldn't inherit all of them.

## Proposed `WORK.md` additions

These are *proposed*, not applied — per the answer at the start of this task, you approve `WORK.md` edits separately. Drafted in the existing rule shape (rule → Why → Example).

### Proposed §1.10. Pre-flight bulk-job estimates against real data.

**Why:** Eyeballed multiplications are reliably wrong. Phase 2 missed embed-runtime estimates by 6×, 12–24×, and 20× across three sub-phases — each time the math was available and not done. The cost of running the chunker (or whatever the unit-of-work is) against real data and counting is tiny; the cost of being surprised mid-run is "user paused at the halfway point."

**Example:** Before estimating a full-corpus embed, run the extractor + chunker against the real data and count the chunks. Multiply by a measured per-chunk rate from a prior pipeline run on the same hardware. "5–10 min" became "60 min" twice in Phase 2 because the chunk-count was eyeballed as one-per-doc when reality was 5–8 per doc.

### Proposed §2.4 generalization (or new §2.9). When fixing a bug in one place, audit peer implementations in the same change.

**Why:** `WORK.md` §2.4 is JOIN-specific ("audit every SELECT list and WHERE clause for newly-ambiguous columns"). The general principle is broader: when a fix lands in one of several near-identical implementations, the parallel ones should be audited in the same edit, not "next time we touch that file." Phase 2 shipped the 503-vs-400 FTS fix in the arxiv router and missed the parallel openalex router. The asymmetry survived a full project retro and four sub-phase retros before being noticed.

**Example:** When fixing operational-vs-syntax error classification in `/arxiv/papers?q=`, audit `/openalex/works?q=` (same shape, same FTS5 JOIN pattern) and fix in the same change. The cost is one extra paragraph in the diff; the cost of forgetting is the bug surviving for months while the rule that prevents it lives next door.

### Proposed §3.8. Defer-the-bulk-job is a ship state, not a failure.

**Why:** Long-running bulk jobs (full-corpus embeds, large rebuilds) can land an endpoint independently of full corpus coverage if the indexer is incremental and the API works against partial data. "Half-finished" is a code state; "shipped against a smaller scope with an idempotent resume path" is a complete state. Phase 2 deferred three of four embeds and shipped all four endpoints.

**Example:** A `/chunks` endpoint backed by a 50-doc sample is fully shipped if (a) the schema is correct, (b) smoke tests pass against the partial corpus, (c) the indexer resumes cleanly without `--reset`, and (d) the deferral is recorded in the retro's carry-over with a one-line resume command. None of those conditions are "we'll finish later" — they're "we shipped a working endpoint and the corpus grows on demand."

### Proposed §1.11. Carry-over items need a disposition after N retros.

**Why:** A carry-over that appears in five retros without movement isn't a TODO — it's either a decision to live with it (which should be stated) or a blocked task (which should name the blocker). The presence-of-carry-over signal is working; what's missing is the resolution loop. The `/health` HTTP status and `kaggle_download.sh` stub have appeared in six retros each.

**Example:** When writing a retro's `Carry-over` section, scan the previous retro's carry-over. Anything that's appeared three times in a row and is still in the same shape: either action it in the next phase, or annotate it as "accepted; documented at <pointer>" in a permanent-limitations doc. The worst state is documented forever, never resolved.
