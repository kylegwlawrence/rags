# Phase 3 — Self-host arxiv ingest

**Date:** 2026-05-19
**Scope:** Port the arxiv ingest pipeline (OAI-PMH harvest, HTML download, structured authors) from `local_wikipedia/arxiv/` into this repo so `data/arxiv/arxiv.db` is no longer a manual `cp` from elsewhere. Add normalized `authors` / `paper_authors` tables preserving `<keyname>` / `<forenames>` / `<affiliation>` per WORK.md §2.1. Rewire `api/routers/arxiv.py` to read authors via the new tables; drop the legacy JSON `papers.authors` column from the API surface. Add a one-shot backfill so the API works against existing data without waiting for a full re-harvest.
**Status:** All three slices shipped across five commits (`b62c5ea`, `4e5b538`, `8205ab0`, `230850d`, `6a17768`). 160/160 tests pass — 145 unit + 15 smoke; broke out from 111 unit at start of Phase 3. Live API verified against the production DB with backfilled author tables (`/arxiv/papers?author=Smith` returns 5 matching papers via the EXISTS join; detail endpoints return full ordered author lists). One operational step is deferred to the user: the full re-harvest + re-download + re-index sequence (hours of wall clock) that produces a fresh DB with structured authors from OAI rather than heuristic-split backfill.

## Summary

Phase 3 is the largest single migration in the project so far. It unblocks deleting `arxiv/` from `local_wikipedia` (the original motivation per `docs/arxiv_migration_plan.md`) and fixes the WORK.md §2.1 author-structure carry-over that's been sitting since Phase 1. The plan landed as three commit-sized slices: parser port → ingest/download CLIs → API rewire + backfill. Each slice had an independent verification step before the next one started.

The cleanest design choice was the **one-shot backfill script** (`scripts/arxiv_normalize_authors.py`). The existing 1238-paper DB has a legacy JSON `authors` column but no `paper_authors` table; the rewired API can't read it. The backfill populates the normalized tables from the JSON column using a heuristic name split — so the API works immediately against existing data, and the multi-hour re-harvest becomes optional rather than blocking. After the re-harvest runs, the backfill becomes obsolete because `arxiv_ingest.py` writes structured authors directly.

Net code: ~1800 lines added across 4 new scripts and 5 new test files, plus ~70 lines modified in `api/routers/arxiv.py` and CLAUDE.md.

## What went well

- **Three-slice plan with explicit transition handling.** Each slice was independently committable and testable. Slice 1 added the OAI parser as a library (no API impact). Slice 2 added the ingest/download CLIs working against a temp DB (still no production impact). Slice 3 rewired the API + ran the backfill (production transition with API uptime preserved). The plan file (`/home/kyle/.claude/plans/staged-enchanting-mountain.md`) was concrete enough to execute from without re-deciding anything mid-stream.
- **Review-pass-after-each-slice discipline.** Both slices 1 and 2 had follow-up hardening commits (`4e5b538`, `230850d`) that fixed real bugs the review caught: empty `<id>` / `<datestamp>` records returning bad rows, autoincrement counter never resetting after `--reset`, `display_name` never updating on dedup hit, empty title/abstract accepted at parse. None of those were urgent, but landing them right after the slice they belonged to kept the carry-over list short for the next slice.
- **Backfill insight (slice 3 design).** The plan as written had a transition gap: rewiring the API would break against existing data until the re-harvest finished (hours). Adding the backfill script — pattern borrowed from `scripts/openalex_normalize_authors.py` — sidesteps that entirely. API works on day 1; re-harvest becomes opt-in.
- **DI-based test mocking, no `respx` dependency added.** `download_papers` accepts a `fetch_fn` argument; tests pass a fake. For the small set of cases that need to exercise the HTTP layer (`fetch_html` 200 / 404 / 503-retry / 429-Retry-After), `httpx.MockTransport` is built-in and gives full coverage. Net new test deps: zero.
- **Smoke run between slice 2 and slice 3.** Ran the new ingest against a 1-day OAI window against a temp DB before doing any production work. 2 papers harvested; structured authors written; 1 HTML body downloaded successfully. Caught nothing wrong but validated the full pipeline before touching `data/arxiv/`.
- **Schema design held up across slices.** The `authors` / `paper_authors` / `ingest_state` tables were specified in slice 2 and didn't change for slice 3. No schema migration mid-Phase.
- **Plan-mode discipline carried.** Three plan-mode entries this session (full Phase 3 plan, slice 3 transition question via AskUserQuestion, slice 1 clarification). No mid-execution scope drift.

## What went wrong / what I learned

- **Slice 3 transition state wasn't in the plan.** I had to surface "what does the API do during the rebuild" as a clarifying question mid-slice. Three options (rewire-only / backfill / atomic rebuild); user picked backfill. The plan file should have addressed this explicitly. **Lesson:** when a code change requires a long data migration, name the transition state explicitly in the plan — what does the system look like during it, and how do users keep working?
- **Test fixture had to `ALTER TABLE add column` to simulate legacy data.** The new schema doesn't have `papers.authors`, but `test_arxiv_normalize_authors.py` needs that column to seed the backfill input. The fixture creates the new schema then ALTERs in the legacy column for the test. Ugly but isolated to one test file. **Lesson:** when porting legacy-to-new, the test fixture has to model both states; if there's a third migration in the future, this pattern repeats.
- **Real test catches.** `test_empty_authors_column_handled` caught a TypeError when `json.loads("null")` returns None and we try to iterate it. `test_whitespace_stripped` caught the trailing-space bug in `rsplit` when input has internal double-spaces. Both fixed before the commit. The tests existed because of the review-after-each-slice discipline above.
- **HTTP download is wall-clock heavy.** arxiv's 3 s rate limit means 1000 HTML bodies takes ~50 minutes. The full re-harvest after slice 3 will run several hours when the user kicks it off. Worth measuring the actual chunks-per-doc when it finishes and updating CLAUDE.md's runtime estimate.
- **`httpx.Client` subclass mocking is verbose.** ~15 lines per test class to set up MockTransport. If we add more HTTP-mocked tests, the right move is extracting a `MockClientFactory` helper to `tests/conftest.py`. Not worth doing for the current four callers in slice 2.
- **`sqlite_sequence` is created lazily.** Discovered when writing the autoincrement-reset fix: `DELETE FROM sqlite_sequence` raises if the table doesn't exist (no INSERTs have fired AUTOINCREMENT yet). Required a `sqlite_master` existence check first. SQLite docs don't make this prominent.
- **Mid-stream `ARXIV_OAI_EMAIL` → `ARXIV_EMAIL` rename.** I named the env var `ARXIV_OAI_EMAIL` in slice 1 then needed to share it with the download script in slice 2. Renamed to `ARXIV_EMAIL` early in slice 2. Cheap fix (one commit) but a forward-looking naming pass during slice 1 would have avoided it. **Lesson:** when an env var or constant is going to be reused by multiple modules in a planned slice sequence, name it for the broadest use case from the start.

## Decisions worth remembering

- **Backfill script for transition handling.** When a new schema is incompatible with existing data and the rebuild is expensive, ship a one-shot backfill that converts old → new format on the existing DB. Lets the new code land without breaking the old data. The backfill is disposable (deleted on next rebuild) but pays for itself in API uptime during the migration.
- **Author dedup key is `(keyname, forenames, affiliation)`, not display_name.** Same human at two affiliations = two distinct rows in `authors`. Same name + same affiliation = one row, even if display_name diverges (suffix added on a later record updates the existing row in place). Matches the openalex pattern and avoids the credentialed-suffix fragmentation that bit openalex pre-normalization.
- **`papers.html_content` is owned by the downloader, not the ingester.** A re-harvest of metadata must never clobber an already-downloaded HTML body. Encoded in `arxiv_ingest._PAPER_COLS` (excludes `html_content`/`download_status`/`downloaded_at`) and verified by `test_update_preserves_html_content`.
- **HTTP retry stays local, not routed through `rag.retry.with_retry`.** Both `arxiv_oai._fetch_with_retry` and `arxiv_download.fetch_html` honor `Retry-After` response headers, which is a behavior `with_retry`'s exception-only model can't express without a major refactor. The local helpers are ~15 lines each and pay for themselves in arxiv etiquette.
- **Ingest watermark advances only on network harvests, not cache replays.** `ingest_state.last_harvested_date` is only written when the OAI feed was actually queried — replaying a stale cache shouldn't move "what's the newest record we've seen" forward. Mirror of the upstream's behavior and the right call.
- **DI over `respx` for test mocking.** Pass `fetch_fn` / `harvest_fn` into functions that drive HTTP; tests pass fakes. `httpx.MockTransport` covers the genuine HTTP-layer tests (4 cases) without a new dep. Result: 36 new tests across slice 2 + 15 new in slice 3 with zero new pip packages.

## Carry-over

- **Full re-harvest from scratch.** `rm data/arxiv/arxiv.db* data/arxiv/arxiv_rag.db*` then `python scripts/arxiv_ingest.py` → `arxiv_download.py` → `arxiv_index_fts.py` → `arxiv_index_rag.py`. Hours of wall clock; user-driven. Produces structured authors with proper `<keyname>` / `<forenames>` / `<affiliation>` separation (today's data is heuristic-split via the backfill). After the rebuild, the legacy `papers.authors` JSON column disappears and `scripts/arxiv_normalize_authors.py` becomes obsolete.
- **Delete `arxiv/` from `local_wikipedia`.** Manual step in that repo. Pre-condition: verify the wiki frontend doesn't import from it any more, and confirm the new pipeline here runs end-to-end after the re-harvest. The original migration motivation realized.
- **Re-measure arxiv chunks-per-doc after re-harvest.** CLAUDE.md's runtime estimate (~25-40 min, ~1.3 chunks/doc) is for title+abstract only. Full-body chunking from rendered HTML will push chunks/doc up significantly. Sample 30 papers post-rebuild and update.
- **Future scope: suffix as part of author dedup key.** Currently `<suffix>` folds into `display_name` only; the structured columns don't carry it. Two papers with the same person + suffix vs no-suffix would dedup correctly today (`UPDATE display_name` on the dedup hit), but if some retrieval angle needs to query authors by suffix, the schema would need `(keyname, forenames, suffix, affiliation)` UNIQUE.
- **Possible cleanup: drop `scripts/arxiv_normalize_authors.py` after re-harvest.** The backfill is a one-shot helper for archaeology; after the next full rebuild, it's never useful again. Could delete then, or leave as a documented legacy migration tool. Mild preference for delete to keep the script list focused on the actual pipeline.
- **Phase 4 (wiki RAG separation) is the natural next code direction.** Same pattern: port a subsystem out of `local_wikipedia` and into this repo with proper schema design. No date attached.

## Proposed `WORK.md` additions

### Proposed §1.12. Long migrations need a transition-state plan, not just an end-state plan.

**Why:** A plan that describes the final schema and final code doesn't tell you what the system looks like during the migration. If the transition is short (minutes), this doesn't matter. If it's long (hours of data rebuild, multi-day re-index), the system has to keep working — and "API broken until rebuild finishes" is a real outage. Phase 3 slice 3 had to surface this as a mid-execution clarifying question because the original plan didn't address it; the answer (add a one-shot backfill script) made the rewire safe to land without coordinating an outage. The transition plan should name what users see during the migration and how the system stays available.

**Example:** Phase 3's slice 3 rewires the arxiv API to read authors from a new normalized table. Existing data only has the legacy JSON column; rebuilding takes hours. Without a transition story, the API would 500 from commit time until rebuild completes. The fix was a separate backfill script that converts existing data to the new schema in place, so the rewire lands and works immediately while the user can opt into the full rebuild later. The plan should have proposed this; instead it surfaced as a mid-slice question.
